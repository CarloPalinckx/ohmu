# Security Vulnerabilities — BOLA

BOLA (Broken Object-Level Authorisation, CWE-639) is a class of vulnerability where an API accepts a user-supplied resource ID without verifying the requesting user owns that resource. Juice Shop intentionally ships several BOLA issues as training targets.

## Affected Endpoints

| Endpoint | Vulnerability | CWE |
|---|---|---|
| `GET /rest/basket/:id` | No ownership check on `:id` | CWE-639 |
| `PUT /api/BasketItems/:id` | No ownership check; any user can update any item | CWE-639 |

## How the Vulnerability Works

### `GET /rest/basket/:id`

The handler fetches `Basket.findByPk(req.params.id)` and returns it directly. It never compares `basket.UserId` to `req.userId`. Any authenticated user can read any basket by incrementing the ID in the URL.

**Missing check (conceptual fix):**

```ts
const basket = await Basket.findByPk(req.params.id)
if (!basket || basket.UserId !== req.userId) {
  return res.status(403).json({ error: 'Forbidden' })
}
```

### `PUT /api/BasketItems/:id`

This route is auto-generated by `finale-rest`. No ownership milestone hook is registered, so any authenticated user can `PUT /api/BasketItems/42` to change the quantity (or other fields) of basket item 42, regardless of who owns it.

**Missing check (conceptual fix using a finale milestone hook):**

```ts
basketItemResource.update.auth(async (req, res, context) => {
  const item = await BasketItem.findByPk(req.params.id, { include: Basket })
  if (!item || item.Basket.UserId !== req.userId) {
    res.status(403).json({ error: 'Forbidden' })
    return context.stop
  }
  return context.continue
})
```

## Ownership Data Model

```
User.id  ←──  Basket.UserId
               Basket.id  ←──  BasketItem.BasketId
```

- To authorise access to a `Basket`: `basket.UserId === req.userId`
- To authorise access to a `BasketItem`: `item.Basket.UserId === req.userId` (requires joining through `Basket`)

## CTF Challenge Hooks

The scoring system calls `challengeUtils.solveIf` when the exploit is detected. The checks fire **after** the vulnerable response is already served — they do not block the attack. See [[Challenge Scoring]] for how this pattern works.

## Watch Out For

- JWT signature validity (`isAuthorized`) does **not** imply ownership. A valid token only proves the caller is a known user — not that they own the requested resource.
- `req.userId` comes from `security.appendUserId()`. If that middleware is absent from the chain, `req.userId` will be `undefined` and any ownership comparison will silently fail permissively.
- The `finale`-generated routes apply auth middleware at registration time differently from hand-written routes. Check both the `server.ts` registration block and any milestone hooks when auditing `/api/*` routes.
