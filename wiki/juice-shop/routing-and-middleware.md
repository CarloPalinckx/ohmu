# Routing and Middleware

All Express wiring lives in `server.ts`. Routes follow a two-layer pattern: a middleware chain validates and decorates the request, then a route handler performs the business logic.

## Route Registration Pattern

```ts
// Typical pattern in server.ts
app.get(
  '/rest/basket/:id',
  security.isAuthorized(),       // 1. Check JWT is present and valid
  security.appendUserId(),       // 2. Inject userId into req object
  require('./routes/rest/basket') // 3. Business logic handler
)
```

The middleware chain is explicit for each route — there is no global "require auth for everything under /rest" catch-all.

## Key Middleware

### `security.isAuthorized()`

- Thin wrapper around `express-jwt`.
- Verifies the `Authorization: Bearer <token>` header against the RS256 public key.
- Populates `req.auth` with the decoded JWT payload on success.
- Returns `401` if the token is missing or invalid.

### `security.appendUserId()`

- Runs **after** `isAuthorized`.
- Looks up the authenticated token in the `security.authenticatedUsers` in-memory map.
- Appends the resolved `userId` (and `bid` basket ID) onto the `req` object for downstream handlers to consume.
- Purpose: provide a stable, verified user identity that handlers can trust without re-decoding the JWT themselves.

## REST vs API Routes

| Prefix | How registered | Handler location |
|---|---|---|
| `/rest/*` | Manually in `server.ts` with explicit middleware chains | `routes/rest/<name>.ts` |
| `/api/*` | Auto-generated by `finale-rest` from Sequelize models | No explicit file; configured inline in `server.ts` |

`finale-rest` auto-generates `GET`, `POST`, `PUT`, `DELETE` endpoints for every model registered with it. Access control on `/api/*` routes is applied differently — through `finale` milestone hooks rather than Express middleware.

## Watch Out For

- Not all `/rest/*` routes apply `isAuthorized`. Some are intentionally public (e.g. product listing). Always check the registration call in `server.ts` to confirm which middleware is attached to a given route.
- `/api/*` routes generated by `finale` do **not** automatically enforce ownership — they will serve or mutate any record by ID. Ownership checks must be added via `finale` milestone hooks, and several are intentionally missing (see [[Security Vulnerabilities — BOLA]]).
- `security.appendUserId()` depends on `security.authenticatedUsers` being populated at login time. If a token is valid but was never logged in through the normal flow, `appendUserId` may not find a matching entry.
